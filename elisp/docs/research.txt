



So, I have a CLI app that integrates with the shell to change between projects.  The app, in Rust, maintains a database of nicknames to paths (or setup file).  Since the Rust code is a subprocess it cannot change the cwd, so it communicates via exit codes to the shell what to do (source a file, cd, etc.).  This works nicely in iTerm2 on Mac with .zshrc (and also should work on other terminals, on Linux OSs, with .bashrc--not fully tested).  However, when I use this within an emacs shell, and the zsh changes dirs, Emacs's shell autocomplete still thinks it is in the old pwd.  So I need a way to refresh the emacs state when running the commands inside emacs shell.  As a workaround, I use M-x dirs to resync the current dir after a change in projects.  Alternatively it would be nice to have an elisp library that interfaces with the Rust app (via CLI calls?) to create a new named-shell per project, so when I switch projects it creates a new shell, runs the shell alias that uses the Rust CLI to find the path to switch to, switches to that path, and syncs emacs autocomplete (or perhaps get the path from Rust CLI and open a shell at that path?

You have two related requirements:

Fix directory tracking inside Emacs when your shell changes directories indirectly.

Design a cleaner Emacs integration so project switching feels native instead of fighting shell-mode.

The key architectural point is this:

Your Rust CLI is doing the right thing by not trying to mutate the parent shell process.

The problem is really in Emacs process state and directory bookkeeping, not in Rust.

In Emacs, the best solution depends heavily on which shell integration you are using:

shell-mode

term / ansi-term

vterm

eshell

shell-mode is the most fragile here, because it does command sending and directory tracking through Emacs process filters and heuristics. If your prompt or shell behavior is unusual, Emacs can lag behind the real shell cwd.

What is happening
High-level design issue
Your Rust app returns instructions to the shell, and the shell executes cd or source ....

That means:

the real shell process changes cwd

but Emacs shell buffer metadata may not get updated

so completion, default-directory, and some commands still behave as though the old path is active

In Emacs terms, default-directory in the shell buffer is stale.

This is why M-x dirs or similar manual refresh helps: it forces Emacs to reconcile with the shell state.

Best approaches
Option A — Stop relying on in-shell cd tracking, and let Emacs create/switch shells at the target directory
This is the cleanest design.

Instead of:

user runs shell alias

alias calls Rust

Rust tells shell to cd

Emacs tries to notice afterward

do this:

Emacs asks Rust for the target path

Emacs creates or switches to a shell buffer whose default-directory is already that path

Emacs launches shell there

Why this is better
Benefits
Emacs and shell start in agreement

completion works immediately

no fragile prompt parsing

easy to create one shell per project

simpler cross-platform story

Risks / downsides
requires some Elisp glue

shifts some UX from shell aliases into Emacs commands

existing terminal workflow remains separate

For Emacs integration, this is the approach I recommend first.

Option B — Keep current alias-based workflow, but explicitly notify Emacs after every directory change
This can work, but it is more brittle.

You would need some way for the shell to tell Emacs:

“My cwd is now X; update buffer-local default-directory.”

That is possible, but the implementation varies by shell mode and can get awkward.

Pros
preserves your current shell alias UX

minimal Rust changes

Cons
more mode-specific

more debugging pain

more likely to break across shell-mode, term, vterm, TRAMP, remote shells, unusual prompts, etc.

Option C — Use vterm or eshell as the main integration target
If you are willing to adjust your Emacs shell setup, this may be the most robust long term.

vterm
very good terminal emulation

shell behavior is closer to real terminal behavior

generally better than shell-mode for modern interactive shell usage

eshell
Emacs-native

can integrate very cleanly with Elisp and project workflows

but differs from real zsh/bash semantics, so it may not fit your CLI alias model

If your workflow is shell-heavy and terminal-authentic, vterm is more attractive than shell-mode.

Recommended architecture
I would structure this as two layers:

Layer 1 — Rust CLI gains a pure query mode
Your Rust app should expose something like:

proj path <nickname>
or

proj resolve <nickname>
and print only the canonical target directory.

Possibly also:

proj env <nickname>
proj setup <nickname>
proj shell-init <nickname>
if there is setup-file logic beyond just cd.

The idea is that Emacs should be able to call Rust in a non-shell-mutating mode and receive structured output.

Good output options
Simple
Just print the path:

/Users/mike/src/foo
Better
Print JSON:

{"path":"/Users/mike/src/foo","setup_file":"/Users/mike/.config/proj/foo.zsh"}
JSON is better if you expect this to grow.

Layer 2 — Emacs command creates/switches named shell buffers per project
Example workflow:

M-x my-proj-shell

prompt for nickname

Emacs runs proj resolve <nickname>

Emacs gets path

Emacs opens or reuses *proj:foo*

buffer default-directory is set to that path

shell starts there

This solves your autocomplete problem because Emacs never has stale directory state.

Nonfunctional requirements to keep in mind
Reliability
Avoid prompt-parsing hacks where possible.

Portability
Keep Rust CLI as the source of truth for nickname → path/setup mapping.

Extensibility
Return structured data from Rust if you may later support:

env vars

custom shell init fragments

project type

preferred shell

project-local commands

Emacs UX
Prefer predictable named buffers:

*proj:foo*

*proj:bar*

This makes switching much nicer than repeatedly mutating one shell.

This is the simplest and probably best first milestone.

Rust

Good fit when

Good fit when

Design 3: Emacs bypasses shell alias entirely for project switching
This is the most Emacs-native.

You retain shell aliases for external terminals, but inside Emacs you use dedicated Elisp commands.

That means two frontends over the same Rust backend:

shell alias frontend

Emacs frontend

This is probably the right separation of concerns.

Suggested Elisp approach
Here is a compact proof-of-concept for shell-mode.

Basic version: open or reuse a named shell at the Rust-resolved path
(defun my-proj--call-cli (&rest args)
  "Call project CLI and return trimmed stdout, or signal an error."
  (with-temp-buffer
    (let ((status (apply #'process-file "proj" nil (current-buffer) nil args)))
      (unless (eq status 0)
        (error "proj failed: %s"
               (string-trim (buffer-string))))
      (string-trim (buffer-string)))))

(defun my-proj-resolve-path (nickname)
  "Resolve NICKNAME to a directory path using the Rust CLI."
  (let ((path (my-proj--call-cli "resolve" nickname)))
    (unless (file-directory-p path)
      (error "Resolved path is not a directory: %s" path))
    (file-name-as-directory (expand-file-name path))))

(defun my-proj-shell-buffer-name (nickname)
  (format "*proj:%s*" nickname))

(defun my-proj-shell (nickname)
  "Create or switch to a shell buffer for project NICKNAME."
  (interactive "sProject nickname: ")
  (let* ((dir (my-proj-resolve-path nickname))
         (bufname (my-proj-shell-buffer-name nickname))
         (buf (get-buffer bufname)))
    (if (buffer-live-p buf)
        (pop-to-buffer buf)
      (let ((default-directory dir))
        (shell bufname)
        (with-current-buffer bufname
          (setq-local default-directory dir)
          ;; Optional: rename buffer again if shell renamed it
          (rename-buffer bufname t))
        (pop-to-buffer bufname)))))
This already gives you:

a named shell per project

correct default-directory

working completion from the start

Improved version: if shell already exists, resync it to the project directory
If you want existing project shell buffers to be reusable and re-centered when needed:

(defun my-proj-shell-sync (nickname)
  "Switch to the project shell for NICKNAME, creating it if needed.
If it already exists, send a cd command and update Emacs state."
  (interactive "sProject nickname: ")
  (let* ((dir (my-proj-resolve-path nickname))
         (bufname (my-proj-shell-buffer-name nickname))
         (buf (get-buffer bufname)))
    (if (not (buffer-live-p buf))
        (let ((default-directory dir))
          (shell bufname)
          (with-current-buffer bufname
            (setq-local default-directory dir)
            (rename-buffer bufname t))
          (pop-to-buffer bufname))
      (pop-to-buffer buf)
      (with-current-buffer buf
        (setq-local default-directory dir)
        (goto-char (point-max))
        (insert (format "cd %s" (shell-quote-argument dir)))
        (comint-send-input)))))
This explicitly updates Emacs’s idea of default-directory before sending cd.

That helps for shell-mode.

What about fixing your current alias workflow inside shell-mode?
You can try to repair it, but I would treat that as secondary.

Why autocomplete gets stale
In shell-mode, Emacs often updates default-directory by recognizing directory changes in the shell output/prompt. If your custom flow changes directories in a way Emacs does not notice, completion remains tied to the old directory.

Possible fixes
1. Ensure shell prompt integration is standard enough
If your prompt emits recognizable cwd information, shell-dirtrack-mode may work better.

Check whether shell-dirtrack-mode or explicit dirtrack-mode is active in that shell buffer.

2. Add explicit pwd after the project-switch command
Sometimes sending pwd after the shell cd gives Emacs enough output to resync, depending on mode setup.

3. Write an Emacs wrapper command around your shell alias
Instead of typing the alias directly, define an interactive Elisp command that:

sends the alias command

then updates default-directory itself based on a separate Rust resolve call

This is a reasonable compromise.

Example:

(defun my-proj-shell-command (nickname)
  "Run the existing shell alias flow, but also resync Emacs."
  (interactive "sProject nickname: ")
  (let ((dir (my-proj-resolve-path nickname)))
    (setq-local default-directory dir)
    (goto-char (point-max))
    (insert (format "pj %s" nickname)) ;; your alias here
    (comint-send-input)))
This is a useful transitional approach.

Recommended architecture
Layer 1: Emacs calls the real binary, not shell aliases
In Elisp, do not call hlpj, chpj, etc. as shell aliases.

Instead call the actual program, likely something like:

or whatever the Rust binary is named.

Why this is better:

aliases are shell-dependent

Emacs process-file / call-process does not automatically expand zsh aliases

direct binary invocation is more portable and reliable

So your Emacs command pjmai-change would internally run:

pjmai change <name>
not:

The aliases remain great for interactive terminal usage, but Emacs should target the stable program interface.

Suggested Elisp skeleton
Here is a compact starting point.

(defgroup pjmai nil
  "Emacs integration for pjmai-rs."
  :group 'tools
  :prefix "pjmai-")

(defcustom pjmai-program "pjmai"
  "Path to the pjmai executable."
  :type 'string)

(defcustom pjmai-buffer-name "*pjmai*"
  "Default output buffer for pjmai commands."
  :type 'string)

(defun pjmai--call (&rest args)
  "Run pjmai with ARGS and return trimmed stdout.
Signal an error if the command fails."
  (with-temp-buffer
    (let ((status (apply #'process-file pjmai-program nil (current-buffer) nil args)))
      (let ((output (string-trim (buffer-string))))
        (if (eq status 0)
            output
          (error "pjmai failed: %s" output))))))

(defun pjmai--display-command (buffer &rest args)
  "Run pjmai with ARGS and display output in BUFFER."
  (let ((buf (get-buffer-create buffer)))
    (with-current-buffer buf
      (setq buffer-read-only nil)
      (erase-buffer)
      (let ((status (apply #'process-file pjmai-program nil buf nil args)))
        (goto-char (point-min))
        (special-mode)
        (unless (eq status 0)
          (error "pjmai command failed; see %s" buffer))))
    (pop-to-buffer buf)))

(defun pjmai-help ()
  (interactive)
  (pjmai--display-command "*pjmai-help*" "aliases"))

(defun pjmai-list ()
  (interactive)
  (pjmai--display-command "*pjmai-list*" "list"))

(defun pjmai-show ()

My concrete recommendation
Use:

C-c p
as the prefix, and start with:

C-c p h   help
C-c p c   change
C-c p s   show
C-c p l   list
C-c p q   query
C-c p t   context
C-c p H   history
C-c p x   exports
C-c p g l group list
C-c p g s group show
C-c p g p group prompt
This is idiomatic, scalable, and easy to learn.

One refinement I would strongly suggest
Add a stable machine-readable CLI command for Emacs, something like:

pjmai query <name> --format json
or

pjmai resolve <name> --format json
with fields like:

project name

root path

setup file if any

current group

maybe a display label

That will make your Emacs integration much more robust than scraping human-readable output.

Bottom line
Yes — a prefix keymap for pjmai-rs is the right direction.

I would implement it as:

prefix: C-c p

subcommands: plain mnemonic letters, not ctrl-letters

backend: call the real pjmai binary, not shell aliases
